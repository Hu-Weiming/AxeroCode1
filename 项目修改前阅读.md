本文档的目标是让后续任何一次修改都可以先读这一份，再决定去看哪些代码，而不是每次都从根目录重新扫描一遍。

项目根目录：`D:\AeroCode1`

## 1. 系统整体功能说明

AxerCode 是一个本地优先、Windows-first 的 AI 编程助手，目标形态对标 Claude Code / Codex 一类工具，目前已经形成三条使用路径：

- `CLI`：本地终端交互，带 REPL、slash commands、计划模式、上下文状态。
- `Web UI`：本地 Spring Boot 服务 + 静态页面，通过 SSE 流式输出对话。
- `Desktop`：当前真实可用的是 `jpackage` 打包链和 desktop preview；`Pake` 路线已进仓库，但仍属于后续壳层路线，不是当前核心运行时。

当前代码里已经真实落地的核心能力：

- 本地 Ollama 模型调用，默认模型是 `qwen2.5:7b`
- 流式输出（CLI 和 Web 都是增量流式，不再一次性整段返回）
- Tool Use / Function Calling
- 反思修正（Reflection）
- SQLite 会话持久化
- CLI 的 `/plan`、`/focus`、`/checkpoint`、`/restore`、`/diff`、`/branch`
- Web 端的 `Plan Mode` 切换
- Web 端 `BTW`：在 agent 正在执行一轮会话时，允许用户补充一句，这条消息会在下一轮 provider 请求前注入

需要特别记住的“真实现状”：

- 当前真正实现的 provider 只有 `Ollama`
- 仓库里有 OpenAI-compatible / Anthropic 风格的设计与抽象说明，但没有完整的具体 provider 实现
- 当前的上下文管理是“滑动窗口裁剪”为主，不是完整的 summary-based context compaction
- `Spring AI` 不在当前实际依赖里

## 5. 代码阅读优先级

不要先扫全仓。推荐顺序如下：

1. `README.md`
2. `pom.xml`
3. 选当前改动入口
4. 再下钻到 agent / provider / tools / storage

按问题类型选择入口：

- 改 CLI 行为：先看 `axercode-cli`
- 改 Web 对话 / SSE / Plan Mode / BTW：先看 `axercode-server`
- 改工具循环 / tool round / reflection / BTW 下一轮注入：先看 `axercode-agent`
- 改模型请求格式 / 流式协议：先看 `axercode-provider-api` 和 `axercode-provider-ollama`
- 改数据库持久化：先看 `axercode-storage-sqlite`

## 9. 修改前最短路径建议

如果你下次只是要快速进入修改状态，先做这几步就够了：

1. 读本文档
2. 读 `README.md`
3. 根据问题类型直接打开对应模块入口文件
4. 只在必要时再读 `docs/steps` 和更深层源码

读完这四步，一般就不需要再从根目录整仓扫描一次。
